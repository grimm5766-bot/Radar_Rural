Você é um desenvolvedor full-stack sênior. Crie uma primeira versão funcional, em formato MVP, de um sistema de gestão agrícola para acompanhamento de lavoura de soja em uma fazenda de 200 hectares.

O sistema deve ter dois perfis principais:

1. Agrônomo
2. Produtor

Use uma stack simples e moderna, preferencialmente:

* Front-end: React ou Next.js
* Back-end: Node.js com Express ou API Routes do Next.js
* Banco de dados inicial: SQLite ou PostgreSQL com Prisma
* Autenticação simples por perfil de usuário
* Dashboard com gráficos usando uma biblioteca como Chart.js ou Recharts
* Aplicativo responsivo, funcionando bem em computador e celular

Crie uma primeira versão funcional com código organizado, rotas, componentes, modelos de dados e telas principais.

Contexto do sistema:

O objetivo do sistema é permitir que o agrônomo registre informações de campo durante o ciclo da soja, acompanhe ocorrências críticas como pragas e doenças, registre fotos geolocalizadas e envie essas informações para que o produtor visualize um dashboard com previsão de safra, alertas e progresso da lavoura.

Requisitos principais do sistema:

MÓDULO DO AGRÔNOMO

1. Checklist de vistoria

Criar uma tela para o agrônomo registrar uma vistoria de campo.

Campos obrigatórios:

* Talhão
* Data da vistoria
* Dia do ciclo da soja
* Fase fenológica
* Umidade do solo
* Altura média da planta
* Pragas encontradas
* Doenças encontradas
* Plantas daninhas encontradas
* Observações gerais

O sistema deve considerar que o ciclo total da soja é de 120 dias.

2. Registro de amostragem para cálculo de produtividade

Criar campos para o agrônomo informar:

* Plantas por metro
* Vagens por planta
* Grãos por vagem
* Peso estimado de mil grãos, chamado PMG
* Área do talhão em hectares

Com base nesses dados, criar uma estimativa simples de produtividade em kg e sacas por hectare.

Considere que 1 saca de soja possui 60 kg.

Deixe o cálculo em uma função separada para facilitar ajustes futuros.

3. Registro de ocorrências críticas

Criar uma tela ou seção onde o agrônomo possa registrar rapidamente uma ocorrência.

Campos:

* Talhão
* Tipo de ocorrência: praga, doença ou planta daninha
* Nome da ocorrência, por exemplo Ferrugem Asiática
* Gravidade: baixa, média ou alta
* Data da ocorrência
* Descrição
* Status: aberta, em controle ou resolvida

4. Controle de manejo

Permitir que o agrônomo abra um chamado de controle para uma ocorrência.

Exemplo:

* Aplicação de fungicida
* Aplicação de herbicida
* Controle biológico
* Outra ação de manejo

Campos:

* Ocorrência relacionada
* Tipo de manejo
* Data de abertura
* Data de fechamento
* Responsável
* Observações

Quando a data de fechamento for preenchida, o sistema deve calcular automaticamente quantos dias demorou para conter o problema.

5. Captura de fotos geolocalizadas

Criar estrutura para anexar fotos a uma vistoria ou ocorrência.

Para a primeira versão, pode usar upload local ou simulação de upload.

Campos da foto:

* Arquivo da imagem
* Latitude
* Longitude
* Data
* Vistoria ou ocorrência relacionada

No front-end, usar a API de geolocalização do navegador para capturar latitude e longitude quando possível.

6. Funcionamento offline

Implementar uma primeira versão de funcionamento offline no front-end.

Quando o agrônomo estiver sem internet, os dados de vistoria, ocorrência e fotos devem ser salvos localmente usando localStorage ou IndexedDB.

Quando a internet voltar, criar uma rotina de sincronização que envie os dados pendentes para o servidor.

Na primeira versão, pode ser uma sincronização simples com uma fila local de registros pendentes.

MÓDULO DO PRODUTOR

1. Dashboard principal

Criar uma tela principal para o produtor com visão geral da fazenda.

Mostrar:

* Total de hectares
* Número de talhões cadastrados
* Dia atual do ciclo
* Fase atual da lavoura
* Estimativa atual de produtividade
* Quantidade de alertas abertos
* Quantidade de ocorrências críticas

2. Gráfico visual do ciclo de 120 dias

Criar um gráfico de linha ou timeline mostrando os 120 dias do ciclo da soja.

O gráfico deve exibir marcações nos dias em que ocorreram:

* Vistorias
* Ocorrências
* Problemas críticos
* Ações de manejo

Usar cores intuitivas:

* Verde: normal
* Amarelo: atenção
* Vermelho: crítico

3. Previsão de safra dinâmica

A cada nova amostragem registrada pelo agrônomo, o sistema deve recalcular a previsão de safra.

Mostrar:

* Estimativa em kg por hectare
* Estimativa em sacas por hectare
* Estimativa total considerando todos os talhões
* Estimativa total para os 200 hectares

4. Histórico de alertas

Criar uma tela com lista de problemas detectados.

Mostrar:

* Talhão
* Tipo de problema
* Nome da ocorrência
* Gravidade
* Data de abertura
* Data de fechamento
* Tempo de resposta em dias
* Impacto estimado na produção
* Status

5. Resumo de progresso do ciclo

Criar uma barra de progresso mostrando:

* Dia atual do ciclo
* Total de 120 dias
* Porcentagem concluída
* Fase atual da planta

Exemplo:

Dia 45 de 120 — 37,5% concluído — Fase vegetativa.

REGRAS DE NEGÓCIO

1. Divisão por talhões

O sistema deve permitir dividir a fazenda em talhões.

Exemplo:

* Talhão A: 50 hectares
* Talhão B: 150 hectares

Cada vistoria, ocorrência, foto, amostragem e manejo deve estar ligado a um talhão.

2. Vazio sanitário

Criar uma regra para impedir o início de um novo ciclo de soja dentro do período do vazio sanitário da região cadastrada.

Para a primeira versão, considere Goiás como padrão.

Exemplo:

* Bloquear início de ciclo entre junho e setembro.
* Exibir mensagem de erro caso o usuário tente criar um ciclo nesse período.

Deixe essa regra em uma função separada para facilitar alterações futuras.

3. Notificações

Criar uma estrutura inicial para notificações.

Quando o agrônomo registrar uma ocorrência com gravidade alta, o sistema deve criar uma notificação para o produtor.

Na primeira versão, simular o envio por WhatsApp ou push notification apenas registrando a notificação no banco e exibindo no dashboard.

REQUISITOS NÃO FUNCIONAIS

1. Interface simples e direta

O sistema deve ter uma interface limpa, com:

* Fontes grandes
* Cards visuais
* Ícones explicativos
* Cores intuitivas
* Layout responsivo
* Menu lateral ou superior separando Agrônomo e Produtor

2. Desempenho

O dashboard do produtor deve carregar rapidamente.

Evite consultas desnecessárias.

3. Segurança básica

Criar autenticação simples com login e senha.

Cada produtor deve visualizar apenas os dados da própria fazenda.

Cada agrônomo deve registrar dados apenas nas fazendas ou talhões vinculados a ele.

4. Privacidade

Estruturar o banco para separar dados por fazenda, produtor e talhão.

Priorize uma versão limpa, funcional e fácil de apresentar como protótipo acadêmico ou MVP.
